To move **Project Aurelius** from a static RAG implementation to a dynamic **World Model**, the `dream` and `teach` commands act as the "Metabolic Layer." These commands ensure that the knowledge graph ($Oxigraph$) and the logic engine ($Ascent$) stay synchronized with reality.

---

## 1. The `dream` Command: Autonomous Consolidation
The `dream` command is a **non-interactive maintenance pipeline**. It runs as a background process or a scheduled CLI task to resolve "entropy" in the memory palace.

### Core Metabolic Processes
| Process | Engine | Action |
| :--- | :--- | :--- |
| **Entity Resolution** | `z-embed` + `Ascent` | Find entities with vector distance $< 0.12$. Propose a `Merge` fact. |
| **Confidence Decay** | `Rust` (Temporal) | Facts not accessed in 30 days lose $5\%$ confidence. |
| **Lattice Pruning** | `Ascent` | If $AccessLevel$ derives to `None` for a defunct project, move related triples to cold storage. |
| **Contradiction Detection** | `Ascent` | Trigger rules that find `A :- Fact(x), !Fact(x)`. Flag for `teach`. |

### Logic Implementation (Rust/Ascent)
In the "dreaming" state, we add a **Resolution Lattice** to our `ascent!` program:
```rust
ascent! {
    relation potential_alias(EntityId, EntityId, f32); // (ID1, ID2, distance)
    
    // If two entities are close in vector space AND share a managed asset, 
    // they are likely the same entity.
    relation merge_proposal(EntityId, EntityId, &'static str);
    merge_proposal(e1, e2, "semantic_overlap") :- 
        potential_alias(e1, e2, dist),
        manages(e1, asset),
        manages(e2, asset),
        if *dist < 0.10;
}
```

### CLI Execution
```bash
# Analyze all wings and consolidate
$ aurelius dream --all

# Focus "dreaming" on a specific business wing to resolve naming conflicts
$ aurelius dream --wing "client_acme" --aggressive
```

## 2. The `teach` Command: Active Reinforcement
While `dream` is autonomous, `teach` is **Human-in-the-Loop (HITL)**. It uses the principles of **Spaced Repetition** to ensure the agent’s "Core Truths" are accurate.

### The "Socratic" Interaction Flow
1.  **Identification:** Aurelius scans for "High Entropy" facts (Confidence $< 0.5$ or flagged by `dream`).
2.  **Contextual Inquiry:** The agent uses Gemma 4 to formulate a clarifying question based on the `derived_from` source document.
3.  **Validation:** User provides the ground truth.
4.  **Promotion:** The triple is updated with `Core_Truth` metadata (Confidence $1.0$) and a reset TTL.

### Interaction Scenario
**User:** `aurelius teach --wing "agency_ops"`
**Agent:** "I have a conflict. An email from March says *Jordan* manages *Project X*, but a contract from April says *Sam* succeeds *Jordan*. Is *Sam* the current primary lead?"
**User:** "Yes, Sam took over on April 1st."
**Agent:** * Updates: `expires_on(Jordan_Management, 1711929600)`
* Adds: `succeeds(Sam, Jordan)`
* Ascent re-computes: `has_access(Sam, ProjectX_Repo, Admin)`

### CLI Flags & Targets
* `--focus <topic>`: Prioritize teaching on `access`, `products`, or `provenance`.
* `--contradictions`: Only show facts where `Ascent` found a logical clash.
* `--verify-lineage`: Walk through facts derived from a specific document to see if they are still "sane."

## 3. Integration: The Metabolic Architecture



### Data Lifecycle in Aurelius 2.0
1.  **Ingest:** `brv ingest` populates the "Drawers" (verbatim) and extracts initial triples into Oxigraph.
2.  **Reason:** `ascent!` immediately computes the **Lattice Access Model**. If a "Dev" is added to an "Org," they immediately inherit "Write" access in the materialized view.
3.  **Dream:** Periodic `dream` runs merge duplicate entities created by slightly different naming in source docs (e.g., "AWS" vs "Amazon Web Services").
4.  **Teach:** The user runs `aurelius teach` once a week. The agent asks about the $3-5$ most confusing logical gaps discovered during the `dream` phase.

### Why this beats Naive RAG
* **No Redundant Logic:** Gemma 4 doesn't have to "figure out" if Bob can access the server. It just sees a fact: `access(Bob, Server, Write)`.
* **Self-Healing:** If you delete a `source_document`, the `derived_from` logic in `Ascent` can automatically flag all triples created from that file as "Stale" or "Invalid."

Adding Personal CRM capabilities fundamentally changes how your agent interacts with you. It bridges the gap between cold, hard business logic and human empathy. 

When you combine business graphs (Contracts, Obligations) with social graphs (Family, Friends), your agent can execute "high-EQ" logic—like reminding you not to chase an overdue invoice because the client's mother just passed away, or suggesting a warm intro to a prospect through a mutual friend.

Here is how to map human relationships and recurring events into your `LanceGraph` + `ascent` pipeline.

---

### 1. The Personal Ontology (The Primitives)

First, we define the human-centric inputs. These facts are usually extracted by Gemma 4 from emails, texts, or conversational offhand remarks ("*I need to take Friday off for my wife's birthday*").

#### A. Identity & Biology
* **`person(EntityId, Name)`**: Distinct from the `entity_is` catch-all.
* **`born_on(EntityId, Month, Day)`**: Storing month and day as `u8` integers is much easier for recurring logic than a fixed Unix timestamp.

#### B. The Social Graph (Edges)
* **`parent_of(EntityId, EntityId)`**: The base directed edge for hierarchy.
* **`spouse_of(EntityId, EntityId)`**: The base edge for marriage.
* **`friend_of(EntityId, EntityId)`**: The base edge for social ties.
* **`life_event(EntityId, EventType, Timestamp)`**: For one-off human events (e.g., `Marriage`, `Graduation`, `Bereavement`).

---

### 2. Auto-Deriving the Human Network (`ascent` Logic)

Human relationships are heavily implicit. If you tell the database "Alice is Bob's wife," you shouldn't have to also tell it "Bob is Alice's husband." The logic engine handles symmetry and inversion.

```rust
ascent! {
    // --- INPUTS ---
    relation parent_of(EntityId, EntityId);
    relation spouse_of(EntityId, EntityId);
    relation friend_of(EntityId, EntityId);

    // --- DERIVED KINSHIP (Symmetry & Inversion) ---
    relation spouse(EntityId, EntityId);
    relation friend(EntityId, EntityId);
    relation child_of(EntityId, EntityId);
    relation sibling_of(EntityId, EntityId);

    // 1. Spouses and Friends are bidirectional (Symmetric)
    spouse(a, b) :- spouse_of(a, b);
    spouse(b, a) :- spouse_of(a, b);

    friend(a, b) :- friend_of(a, b);
    friend(b, a) :- friend_of(a, b);

    // 2. Children are the inverse of parents
    child_of(child, parent) :- parent_of(parent, child);

    // 3. Siblings share a parent
    sibling_of(a, b) :- 
        parent_of(p, a), 
        parent_of(p, b), 
        if *a != *b;

    // --- TRUST & ACCESS (The "Warm Intro" Rule) ---
    lattice social_trust(EntityId, EntityId, Confidence);

    // Immediate family has high implicit trust
    social_trust(a, b, Confidence(0.9)) :- spouse(a, b);
    social_trust(a, b, Confidence(0.8)) :- parent_of(a, b);
    social_trust(a, b, Confidence(0.8)) :- child_of(a, b);
    
    // Friends have moderate trust
    social_trust(a, b, Confidence(0.6)) :- friend(a, b);

    // Transitive Trust: "A friend of a friend"
    social_trust(a, c, Confidence(conf)) :- 
        friend(a, b), 
        social_trust(b, c, Confidence(c_bc)),
        let conf = 0.6 * c_bc; // Degrades over distance
}
```

### 3. Handling Time: Birthdays & Anniversaries

Handling recurring dates in pure Datalog is notoriously messy because you have to account for leap years and month rollovers. 

**The Rust Solution:** Handle the "calendar math" in standard Rust *before* injecting the facts into `ascent`. When your pipeline wakes up for the day, Rust scans LanceGraph and computes a `days_until` relation for the engine.

```rust
ascent! {
    // --- INPUTS ---
    relation days_until_event(EntityId, &'static str, u32); // (Person, EventType, Days)
    relation client_status(EntityId, &'static str);

    // --- ACTIONS & ALERTS ---
    relation agent_action(ActionType, EntityId, EntityId);

    // Rule: Remind me 7 days before my wife's birthday
    agent_action(ActionType::DraftGiftIdea(person), "Operator", person) :-
        spouse("Operator", person),
        days_until_event(person, "Birthday", days),
        if *days <= 7;

    // Rule: Draft a warm check-in email for an active client whose birthday is tomorrow
    agent_action(ActionType::DraftEmail(client), client, client) :-
        client_status(client, "Active"),
        days_until_event(client, "Birthday", days),
        if *days == 1;
}
```

### 4. High-EQ Business Logic (The Intersection)

The real magic happens when the "System 2" brain evaluates a business problem through a human lens. We can write rules that prevent the agent from making tone-deaf automated decisions.

```rust
ascent! {
    // ... (assuming previous schemas are included)
    relation life_event(EntityId, &'static str, Timestamp);
    relation overdue(EntityId, EntityId, EntityId); // (Invoice, Client, Agency)
    relation active_obligation(EntityId, EntityId, EntityId, Timestamp);

    // --- COMPASSION PROTOCOLS ---
    relation hold_communications(EntityId, &'static str);

    // Rule: If a client (or their immediate family) had a bereavement in the last 30 days, halt automated collections.
    hold_communications(client, "Bereavement Protocol") :-
        life_event(client, "Bereavement", event_time),
        current_time(now),
        if *now - *event_time < 2_592_000; // 30 days in seconds

    hold_communications(client, "Bereavement Protocol") :-
        social_trust(client, family_member, Confidence(c)),
        if *c >= 0.8, // Immediate family threshold
        life_event(family_member, "Bereavement", event_time),
        current_time(now),
        if *now - *event_time < 2_592_000;

    // Override the automated invoice reminder from our previous schema
    agent_action(ActionType::SendInvoiceReminder, client, id) :-
        invoice(id, client, "MyAgency", _),
        overdue(id, client, "MyAgency"),
        !hold_communications(client, _); // The Denial Guard
}
```

### The Output
If Gemma extracts *"Hey, I'll be slow to respond, my father just passed away"* from a client email, it injects `life_event(Bob_Father, "Bereavement", <timestamp>)`. 

The next day, Bob's invoice goes overdue. Instead of the system automatically firing off a "Please pay your invoice" email, the `hold_communications` rule blocks it, and the agent simply flags the invoice for your manual review with the note: *"Held due to Bereavement Protocol."*
