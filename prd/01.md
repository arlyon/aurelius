# Product Requirements Document (PRD): Rust-Recall

## 1. Introduction / Overview
**Rust-Recall** is a fully on-device, high-performance command-line interface (CLI) tool designed for Retrieval-Augmented Generation (RAG). Built entirely in Rust (bypassing heavy Python dependencies), it transforms local filesystems—including complex PDFs, Office documents, and deep codebases—into a highly queryable, locally-hosted knowledge graph. By combining Kreuzberg for structural parsing, LanceDB for hybrid search, and Gemma 4 (e4b) for on-device synthesis, Rust-Recall delivers state-of-the-art recall without sacrificing user privacy or requiring cloud APIs.

## 2. Goals / Objectives
* **Primary Objective:** Deliver a standalone, 100% offline Rust RAG CLI that achieves exceptional retrieval accuracy by preserving document structure during chunking and utilizing zero-dependency local embeddings (zembed-1).
* **Secondary Objective:** Architect the system's core indexing and retrieval pipelines (via Swiftide and LanceDB) so they can eventually be compiled to WebAssembly (Wasm) for serverless deployment on edge networks (e.g., Cloudflare Workers).

## 3. Target Audience / User Personas
* **The "Power Developer":** Needs to index hundreds of local repositories (Rust, Python, TS) and query them for architectural context without uploading proprietary code to OpenAI or Anthropic.
* **The "Deep Researcher":** Manages thousands of local PDFs, academic papers, and Word documents and requires a fast, offline tool to synthesize answers with precise citations back to the exact local file.

## 4. User Stories / Use Cases
* **US1:** As a user, I want to run a single command (`aurelius ingest ./docs`) to recursively parse a directory, respecting my `.gitignore` rules, so I don't index build artifacts.
* **US2:** As a researcher, I want the system to understand the structure of my documents (tables, headers) rather than blindly cutting text at 512 tokens, so the AI doesn't lose the context of complex paragraphs.
* **US3:** As a user, I want to ask a natural language question in my terminal and receive a conversational answer synthesized by a local LLM, complete with inline citations pointing to the files it used as context.
* **US4:** As a power user, I want the search functionality to match exact keywords (like a specific function name) while also understanding the semantic "vibe" of my query.

## 5. Functional Requirements
* **FR1: Document Ingestion & Parsing:** The CLI must recursively scan directories using the `ignore` crate. It must utilize `Kreuzberg` to extract structured Markdown from 90+ non-image formats (PDF, DOCX, CSV, etc.) and perform syntax-aware parsing for 248 programming languages via Tree-sitter. Images/OCR are excluded from the MVP.
* **FR2: Context-Aware Chunking:** The pipeline (via Swiftide) must chunk the Kreuzberg-generated Markdown. Each chunk must automatically receive a prepended "Global Context" header containing the source document metadata to prevent context fragmentation.
* **FR3: Local Vectorization & Storage:** The CLI must generate embeddings locally using the `zembed-1` (4B) model via the `candle` crate. Data must be stored in a local LanceDB instance.
* **FR4: Hybrid Search Execution:** The `search` command must execute a Two-Stage Retrieval using LanceDB:
    * **Stage 1:** Execute LanceDB's native `FullTextSearchQuery` for BM25 keyword matching and standard vector search for Cosine Similarity.
    * **Stage 2:** Merge and rank results using Reciprocal Rank Fusion (RRF).
* **FR5: Local Answer Synthesis & Citation:** The CLI must pipe the retrieved chunks into a locally running Gemma 4 (e4b) model. The generated answer *must* include inline citations (e.g., ``) based on the metadata passed through LanceDB.

## 6. Non-Functional Requirements
* **NFR1: Performance (Latency):** The hybrid search retrieval phase (finding the chunks, before LLM generation) must complete in `< 500ms` for datasets up to 10,000 chunks.
* **NFR2: Hardware Adaptability:** * *High-End (e.g., RTX 2080 Ti):* Full GPU offload utilizing CUDA for max throughput.
    * *Laptop (16GB RAM):* Automatically apply 4-bit quantization (Q4_K_M) for the LLM and INT8 for the embedder, operating strictly within system memory/iGPU limits.
* **NFR3: Privacy & Portability:** The application must compile to a standalone Rust binary with zero Python dependencies and function entirely offline.

8. Success Metrics

    Accuracy: Recall@10 > 0.92 for technical and documentation-based queries.

    Citation Reliability: 100% of LLM-generated responses contain a valid source citation linking to the indexed file.

    Robustness: Ingestion pipeline successfully parses a 1,000+ file repository of mixed formats without fatal crashes from unsupported file types.

9. Open Questions / Future Considerations (Post-MVP)

    Email Integration: Direct ingestion of email archives by downloading and parsing .eml / .msg files.

    Entity Recognition: Extracting specific nouns, people, and technologies during the Kreuzberg phase to tag chunks.

    Fact Database Construction: Isolating standalone "facts" from chunks into a structured relational schema.

    Graph Relationships: Transitioning to GraphRAG by linking entities and facts to understand multi-hop relationships across different documents.
